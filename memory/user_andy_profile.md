## Working style

- Thinks at the product/vision level; wants strategic advice, not just execution
- Comfortable with technical architecture discussions
- Prefers deliberation over hasty decisions on foundational questions — when stakes are high, he wants options with tradeoffs, not single recommendations
- Has deep personal investment in the subject matter (Life Chronicle is partly his own story)
- Engaged with AI tools extensively and sophisticatedly — treat as a knowledgeable peer

## In build sessions (Claude Code)

- Per `~/.claude/CLAUDE.md`: plan-first, small incremental changes, TDD-first, clear commit messages — but Andy explicitly overrides TDD for the local alpha where appropriate
- Will accept a build proposal in 3–5 step plan form, then confirm before execution
- Pauses to question design choices mid-build (good — he's catching things). Stop and discuss when he asks; don't just keep building
- Comfortable letting Claude Code make architectural calls but expects to be briefed on them — e.g. "I'm using Sonnet 4.5 here because…"
- Prefers warm, conversational copy in user-facing UI (the system is intimate); prefers terse, factual reply text in build chat

## How to apply

Frame responses at the product strategy level. Don't over-explain basics. Offer architectural options with tradeoffs rather than single recommendations when stakes are high. In build work, propose plans before changes and surface decisions made along the way. When something feels like it might conflict with prior decisions (PRD, schema, memory files), stop and check rather than assume.
